# ID_ENG_BOM — 생산 BOM

## 확인된 역할

`ID_ENG_BOM`은 생산과 자재 소요에 사용하는 품목 구성정보다. 단순 설계 참고용이
아니라 생산계획·작업지시·자재출고·자동출고·키팅에서 직접 사용한다.

생산 BOM은 외부 시스템 수신본이 아니라 MES가 단독으로 관리하는 기준정보다.
현행 운영 등록·변경 방식은 엑셀 업로드이며 ECO 기능은 사용하지 않는다.
애플리케이션에 남아 있는 `BOM_SYNC` 인터페이스 코드는 정상 기준 경로로 보지 않는다.

## 식별자와 규모

- 기본키: `PARENT_ITEM_CODE + CHILD_ITEM_CODE + DATESET + ORGANIZATION_ID`
- 2026-07-12 라이브 데이터: 9,972건
- 부모 품목: 884개
- 자품목: 2,333개
- 적용 시작일 범위: 2020-08-10 ~ 2026-06-01

## 주요 정보

- 부모·자품목과 단위 소요량
- 적용 시작일·종료일
- 투입 공정과 라인
- 제조유형과 구입유형
- 로스율·스크랩율
- 하위 반제품 전개 여부
- 리비전과 위치정보

## BOM 적용기간

`DATESET`과 `DATEEND`는 생산 BOM 구성관계의 적용 시작일과 종료일이다. 업무 모델은
이 기간을 이용해 동일 상위·하위 구성관계의 과거·현재·미래 BOM을 동시에 관리한다.
기존 BOM을 덮어쓰는 구조가 아니라 적용기간이 다른 버전을 함께 보존하는 구조다.

Oracle `PKG_DESIGN`의 ECO 반영 로직은 신규 BOM 적용 시 동일 구성관계의 기존 행 중
종료일이 신규 시작일 이후인 행을 찾아 `DATEEND = 신규 DATESET - 1일`로 자동
종료한다. 교체·삭제 ECO에도 같은 기간 종료 원칙을 적용한다.

현재 운영에서는 ECO나 적용기간 전환 로직을 사용하지 않는다. 엑셀 업로드 대상인
특정 상위품목의 기존 BOM을 모두 삭제한 뒤, 해당 상위품목의 엑셀 데이터를 다시
올리는 방식으로 구성을 갱신한다. `ID_ENG_BOM` 전체를 삭제하는 것은 아니다.

이 방식은 해당 상위품목의 기존 적용기간 버전도 함께 제거한다. 따라서 스키마는
과거·현재·미래 버전의 동시 관리를 지원하지만 현행 운영 데이터에는 단일 유효 버전만
남는다.

레거시 은성 엑셀 경로는 `ID_ENG_BOM_EXCEL_ES` 입력을
`ID_ENG_BOM_WORKSPACE`에 적재한 뒤 반영하는 구조다.

`ID_ENG_BOM` 자체 INSERT 트리거는 품목코드만 검사한다. 현재 NestJS 엑셀 업로드는
`PKG_DESIGN`을 호출하지 않고 `ID_ENG_BOM`에 직접 INSERT하므로 기존 BOM 종료일
자동 조정과 레거시 워크스페이스 반영 절차가 수행되지 않는다. 이는 현행 운영 방식과
신규 코드의 불일치다. 또한 신규 업로드 코드는 특정 상위품목의 기존 BOM 전체 삭제도
수행하지 않고 중복을 차단한 뒤 신규 행만 추가한다.

2026-07-12 현재 라이브 데이터는 9,972건 모두 유효 상태이고 동일 상위·하위 품목의
기간별 복수 버전은 없다. 이는 현행 데이터가 기간 관리 기능을 적극 사용하지 않는
상태일 뿐, 과거·현재·미래 BOM을 관리하지 않는다는 의미가 아니다.

`REVISION`은 외부 또는 설계 리비전을 참고로 기록하기 위한 정보다. 현재는 사용하지
않으며 BOM의 실제 버전 선택 기준은 적용기간이다. 라이브 데이터는 NULL 9,850건,
`1` 122건이고 동일 구성관계의 복수 리비전은 없다.

## 하위 BOM 전개

`ASSY_EXPLOSION_YN`은 하위 반제품의 BOM을 계속 재귀적으로 전개할지 결정한다.
`Y`는 계속 전개하고 `N`은 해당 하위품목을 하나의 투입 단위로 보아 전개를 중지한다.
현재 라이브 데이터는 모두 `Y`다.

Oracle 패키지와 기존 트리거는 `Y`를 전제로 하지만 현재 백엔드 엔티티 기본값은
`N`이다. 이는 현행 코드 불일치로 기록하며 이 문서 작업에서는 수정하지 않는다.

`WORKSTAGE_CODE`는 하위품목이 투입되는 공정을 참고로 기록하기 위한 정보다. 현재는
사용하지 않으며 라이브 데이터 9,972건 모두 `*`다.

## 레거시 품목 속성 복제

`ITEM_TYPE`과 `LINE_TYPE`은 상위품목 마스터의 제조유형·구입유형을 편의상 BOM에
복제해 둔 레거시 컬럼이다. 별도의 BOM 업무 개념이 아니며 기준값은 항상
`ID_ITEM`이다.

라이브 데이터에서 BOM `ITEM_TYPE`은 조회 가능한 상위품목의 값과 모두 일치하지만,
`LINE_TYPE`은 다수 불일치한다. 이는 복제 구조가 만든 데이터 정합성 문제이며 BOM
값을 기준정보로 해석하지 않는다. 현재 백엔드가 BOM `LINE_TYPE`을 `side`로 매핑하는
것도 실제 의미와 맞지 않는 코드 불일치다.

## 품목 관계

가장 많은 구성은 반제품(`W`) 부모에 원자재(`R`) 자품목이 연결된 경우로
7,967건이다. 모든 현행 행은 `ASSY_EXPLOSION_YN='Y'`다.

표준 업무 용어는 `PARENT_ITEM_CODE=상위품목`, `CHILD_ITEM_CODE=하위품목`이다.
모품목·자품목, 부모·자식은 같은 관계를 가리키는 기존 동의어로만 취급한다.

`ID_ENG_BOM`에는 `ID_ITEM`을 향한 활성 FK가 없으므로 부모·자품목 코드가 품목
마스터와 불일치할 수 있다. 실제 조사에서도 품목구분을 찾지 못한 관계가 존재했다.
