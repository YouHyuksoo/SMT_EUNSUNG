# 장애 학습 정규화 및 규칙 승격 절차

프로젝트 진행 중 발견한 결함, 검증 누락, 잘못된 가정은 수정 코드와 대화에만 남기지 않는다. 반복 가능한 교훈을 정규화해 프로젝트 규칙과 실행 가능한 검증으로 승격한다.

## 1. 적용 시점

다음 중 하나에 해당하면 수정 완료 전에 이 절차를 수행한다.

- 데이터 무결성, 인증, 권한, 테넌트 격리 문제
- 컴파일·테스트는 통과했지만 실제 기능이 실패한 경우
- 같은 유형이 다른 화면이나 모듈에서도 발생할 수 있는 경우
- 잘못된 완료 보고를 유발한 검증 누락
- 프로젝트 문서와 실제 코드 패턴이 달라 발생한 문제
- 사용자가 재발 방지 또는 규칙화를 요구한 경우

단순 오탈자처럼 원인이 해당 코드 한 곳에만 있고 재사용 가능한 교훈이 없으면 회귀 테스트만 추가하고 규칙 승격은 생략할 수 있다.

## 2. 원인 정규화

장애를 다음 네 항목으로 분리해 기록한다.

| 항목 | 질문 |
|---|---|
| 직접 원인 | 어떤 코드·설정·데이터가 실제 실패를 만들었는가? |
| 유입 원인 | 어떤 구현 습관이나 구조 때문에 결함이 들어왔는가? |
| 검출 실패 | 기존 테스트와 검증이 왜 잡지 못했는가? |
| 시스템 원인 | 다른 모듈에서도 반복될 수 있는 공통 조건은 무엇인가? |

증상 자체를 규칙으로 만들지 않는다. 예를 들어 “구매단가 조회 확인”이 아니라 “테넌트 주입이 필요한 컨트롤러는 Guard 적용을 실행 가능한 테스트로 검증한다”로 일반화한다.

## 3. 적용 범위 결정

규칙은 참인 범위보다 넓게 승격하지 않는다.

| 범위 | 판단 기준 | 저장 위치 |
|---|---|---|
| 코드 한정 | 특정 함수·화면에만 해당 | 코드 주석 없이 회귀 테스트 중심 |
| 도메인 한정 | 특정 업무 흐름에 반복 적용 | `docs/business-logics/` 또는 관련 테스트 |
| 프로젝트 공통 | 이 저장소의 여러 모듈에 적용 | `AGENTS.md` 요약 + `docs/standards/` 상세 |
| 사용자 공통 | 기술·프로젝트와 무관하게 모든 작업에 적용 | `ai-instruction-manager`를 통한 사용자 전역 지침 |

프로젝트의 클래스명, 포트, DB 테이블, 인증 구조는 사용자 전역 규칙으로 올리지 않는다.

## 4. 필수 산출물

프로젝트 공통 규칙으로 승격할 때는 다음을 한 묶음으로 처리한다.

1. **중앙 규칙**: `AGENTS.md`에 짧고 명령형으로 추가한다.
2. **상세 표준**: 근거, 적용 조건, 올바른 예시, 진단 순서를 `docs/standards/`에 작성한다.
3. **실행 가능한 강제 장치**: 아래 중 하나 이상을 추가한다.
   - 회귀 단위 테스트
   - Guard·모듈 경계 등의 architecture/structure test
   - lint 또는 정적 검사
   - 인증된 실제 API 검증 명령
4. **완료 기준 갱신**: 기존 검증이 만든 거짓 양성을 차단하는 확인 항목을 추가한다.
5. **실제 재검증**: 수정된 규칙이 현재 장애를 잡고 정상 상태도 통과하는지 red-green으로 확인한다.

문서만 추가한 상태는 규칙화 완료가 아니다.

## 5. 승격 절차

1. 장애를 실제 환경이나 최소 재현으로 확인한다.
2. 직접 원인과 검출 실패를 각각 증거로 확정한다.
3. 수정 전 실패하는 회귀 테스트를 작성한다.
4. 최소 수정 후 테스트와 실제 기능을 다시 확인한다.
5. 반복 가능성을 평가해 적용 범위를 결정한다.
6. 중앙 규칙, 상세 표준, 실행 가능한 강제 장치를 함께 반영한다.
7. 관련 문서의 `sources`와 `verifiedCommit`을 확인한다.
8. 최종 보고에 직접 원인, 시스템 원인, 새 강제 장치, 실제 검증 결과를 구분해 기록한다.

## 6. 규칙 품질 점검

새 규칙은 다음 질문에 모두 답할 수 있어야 한다.

- 어떤 조건에서 적용되는가?
- 어떤 조건에서는 적용되지 않는가?
- 사람이 기억하지 않아도 자동으로 실패를 감지하는가?
- 기존 규칙과 중복되거나 충돌하지 않는가?
- 프로젝트 전용 사실이 사용자 전역 지침으로 새지 않았는가?
- 다음 작업자가 정확한 검증 명령을 실행할 수 있는가?

답할 수 없는 규칙은 추상적인 구호이므로 승격하지 않는다.

## 7. 이번 구매단가 사례의 정규화 결과

| 단계 | 반영 내용 |
|---|---|
| 직접 원인 | Guard 누락과 Oracle bind 객체 재사용 |
| 검출 실패 | 컨트롤러 직접 호출 테스트와 Swagger/HTTP 200 확인만 수행 |
| 프로젝트 규칙 | Oracle DB 기반 화면은 인증된 실제 데이터까지 검증 |
| 상세 표준 | `docs/standards/oracle-db-backed-screen-verification.md` |
| 강제 장치 | Guard 메타데이터 테스트, bind 변경 회귀 테스트, 실제 API·프록시 건수 확인 |

이 형식을 이후 장애의 규칙 승격 기준으로 사용한다.
