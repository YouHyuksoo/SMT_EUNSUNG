## 1. 목표와 범위

OEE 집계·분석·(향후)마감이 신뢰되려면 그 뿌리인 기준정보가 완비·버전관리·검증되어야 한다. 이 플랜은 **OEE가 직접 소비하는 기준정보**에 한정해, 문서 3원칙 중 실효 있는 두 가지(기간버전·마감검증)를 얹는 것을 목표로 한다.

### 범위 포함 (OEE 소비 기준정보)

| 계열 | 기준정보 | 유형 | 현행 상태 |
|------|----------|------|-----------|
| 공정·설비 | 공정 마스터 (`IP_PRODUCT_WORKSTAGE`) | 단순형 | 기존 `master/process` 보강 |
| 공정·설비 | 설비 마스터 (`IMCN_MACHINE`) | 단순형 | 기존 `master/equip` 보강 |
| 공정·설비 | 라인/셀·설비 매핑 | **버전형** | **신규** (매핑 정식화) |
| 공정·설비 | 라우팅 (거래처×모델 공정순서) | **버전형** | 기존 `master/routing` 버전화 |
| 시간 | 월력 (근무일/휴일/특근) | **버전형** | 기존 `master/work-calendar` 버전화 |
| 시간 | 근무시간표 (근무조·휴게·계획정지) | **버전형** | 기존 `shift-pattern` 버전화 |
| 시간 | CT/ST/SETUP 표준시간 | **버전형** | **신규** (트레이서 1순위) |
| 코드 | 무작업/비가공 사유코드 | 단순형 | 기존 `oee/master/reason` 보강 |
| 품목·BOM | 모델 마스터 (생성유형) | 단순형 | 기존 보강 (OEE는 라우팅 키로 참조) |

### 범위 제외 (의도적)

- 수불 전용 기준정보(구매단가·판매단가·창고 수불 거래유형코드·안전재고) — 수불 프로젝트 소관
- **승인(approval) 워크플로우** — 이번 범위 전면 제외(후속 과제)
- **OEE 마감(closing) 프로세스 자체** — 미구현. 본 플랜은 마감이 호출할 검증 API 계약만 제공
- ERP 인터페이스

## 2. 아키텍처 결정 (grill-me 확정)

1. **데이터 범위** — OEE 소비 기준정보만.
2. **3원칙 계층화** — 버전형=기간버전+마감검증, 단순형=일반 CRUD+마감검증. 승인은 전 항목 제외. 마감검증은 전 항목 공통.
3. **버전 모델** — 행-버전 방식으로 통일. 같은 자연키에 적용기간(`VALID_FROM`/`VALID_TO`)이 다른 여러 행이 곧 버전. 개방형 종료는 `DATE '2099-12-31'` 센티넬(기존 BOM 규칙 계승).
4. **as-of 기준일** — 실적 발생일. 각 실적은 그날 유효했던 버전으로 계산되어 과거 OEE를 당시 기준으로 재현.
5. **마감검증** — 재사용 검증 서비스 + API 계약 + "기준정보 준비도 점검" 화면. 마감 자체는 만들지 않음.
6. **리소스 SSOT** — 물리 마스터(설비·라인·공정)를 단일 원본으로 확정. 기존 `oee_resource`(MACHINE/LINE/WORKGROUP 플랫 테이블)는 폐지하고 "물리 마스터 참조 + 매핑"으로 재정의.
7. **라인/셀 축** — `PLANTS` 조직트리(공장/공장동/라인/셀)를 OEE 라인/셀 집계 축의 SSOT로. `IP_PRODUCT_LINE`은 SMT 운영 속성 소스로 라인 노드에 연결(폐지하지 않음).
8. **딜리버리** — 수직 트레이서. CT/ST/SETUP 한 종을 end-to-end로 먼저 완성해 아키텍처 검증 후 패턴 복제.
9. **CT/ST/SETUP 원본** — 산재 컬럼을 통합한 **전용 표준시간 마스터를 `모델×공정×라인/셀` 키로 신설**. 기존 `ROUTING_PROCESSES`·`PROCESS_MAPS`·`PROCESS_CAPAS`의 `STD_TIME`은 이관/참조로 정리, `PROD_RESULTS.CYCLE_TIME`=실측(원본 아님), `ITEM_MASTERS.TACT_TIME`=품목 참고값.
10. **라인 대응** — `IP_PRODUCT_LINE` ↔ `PLANTS`는 공통키(`LINE_CODE`+`ORGANIZATION_ID`)로 매칭. 비어있는 PLANTS 라인/셀은 레거시 라인을 **1회 시드승격**하고 셀은 PLANTS에서 신규 정의. 이후 라인 SSOT=PLANTS, 운영 속성(UPH·CAPACITY)=IP_PRODUCT_LINE.
11. **셀 정의** — OEE 집계 단위 = `PLANTS` CELL 노드 자체. **후공정 셀은 설비 매핑 없이 셀 노드에 실적/무작업 귀속**, 설비→노드 매핑은 설비가 명확한 공정(SMT·ICT·WAVE 등)에만. 억지 세분화 금지(문서 11절).
12. **준비도 심각도** — "OEE 수치 산출 불가 = BLOCK, 분석 품질 저하 = WARN" 원칙으로 규칙별 기본 분류표를 코드에 명시(설정 조정 가능). 향후 마감 게이트가 BLOCK만 차단.

## 3. 공통 데이터 모델

### 3.1 버전형 공통 컬럼 규칙

모든 버전형 테이블은 다음 공통 컬럼을 갖는다.

- `VALID_FROM DATE NOT NULL` / `VALID_TO DATE NOT NULL` (개방형 = `2099-12-31`)
- `STATUS` — `ACTIVE`(현재적용) / `HISTORY`(과거버전). 승인 제외이므로 대기/미승인 상태는 두지 않음. `미등록`은 행 부재로 표현.
- `UPDATED_BY` / `UPDATED_AT` (수정자·수정일)

과거 버전은 삭제하지 않고 `HISTORY`로 보존한다(원칙 ① 화면 구현).

### 3.2 공통 유틸(백엔드 공유 모듈)

버전형 도메인이 공유하는 두 유틸을 신규 작성한다.

- **as-of 유효행 선택** — `(naturalKey, asOfDate) → 유효 1행`. `VALID_FROM <= asOf < VALID_TO AND STATUS='ACTIVE'`.
- **기간 겹침 검증** — 같은 자연키의 적용기간이 겹치지 않도록 저장 시 검증. 신 버전 등록 시 직전 `ACTIVE` 행의 `VALID_TO`를 자동 절단하고 `HISTORY` 전환.

### 3.3 라인/셀 축(PLANTS) 정비와 설비 매핑

- **라인/셀 축 = `PLANTS` 조직트리**(공장/공장동/라인/셀). 현재 라인/셀 시드가 없어 정비 선행.
- **PLANTS 라인 시드승격** — `IP_PRODUCT_LINE`의 라인을 공통키(`LINE_CODE`+`ORGANIZATION_ID`)로 PLANTS LINE 노드에 1회 적재. 셀 노드는 PLANTS에서 신규 정의. 운영 속성(UPH·CAPACITY 등)은 IP_PRODUCT_LINE이 라인 노드에 연결돼 유지.
- **집계 단위** — SMT=라인 노드, 후공정=CELL 노드. **CELL 노드 자체가 리소스**이며, 후공정 셀은 설비 매핑 없이 셀에 실적/무작업 귀속.
- **설비 매핑 (신규, 버전형)** — 설비가 명확한 공정에만 적용. `MACHINE_CODE` → `PLANT_CODE/SHOP_CODE/LINE_CODE/CELL_CODE` + `VALID_FROM/TO`. 설비 구분 약한 공정은 라인/공정 단위까지만(억지 세분화 금지, 문서 11절).

### 3.4 CT/ST/SETUP 표준시간 마스터 (신규 통합, 버전형 · 트레이서 1순위)

- **성격** — 신규 생성이 아니라 **산재 컬럼의 권위 원본 통합**. `STD_TIME`이 `ROUTING_PROCESSES`·`PROCESS_MAPS`·`PROCESS_CAPAS` 3곳에 중복 존재.
- 키: `모델(품번) × 공정 × 라인/셀`
- 값: `CT(s)`, `ST(s)`, `SETUP(분)`, `VALID_FROM/TO`, `STATUS`, `UPDATED_BY/AT`
- 사용처: 성능율(CT), 계획(ST), 교체손실(SETUP) — 문서 12절
- **정리 원칙** — 기존 산재 `STD_TIME`/`SETUP_TIME`은 이 마스터로 이관 또는 참조로 대체. `PROD_RESULTS.CYCLE_TIME`=실측(원본 아님), `ITEM_MASTERS.TACT_TIME`=품목 참고값(원본 아님).

## 4. 마감검증 & 준비도 점검

### 4.1 검증 규칙 엔진

- 인터페이스: `MasterDataValidator { validate(asOfDate, scope): ValidationIssue[] }`
- 각 기준정보 도메인이 자기 검증기를 레지스트리에 등록(신규 기준정보는 검증기만 추가).
- 대표 규칙: 단가/CT/ST 미등록, 라우팅 공백, 라인/셀/설비 매핑 누락, 월력 미등록, 근무시간 미등록, 모델 생성유형 공란(→백플러시 보류).

### 4.2 API 계약 (향후 OEE 마감이 소비)

- `GET /oee/master/readiness?asOf=YYYY-MM-DD&scope=...` → `ValidationIssue[]` + 종합 판정(OK/BLOCK).
- 마감이 생기면 동일 엔진을 게이트로 호출(하드코딩 없이 재사용).

### 4.3 준비도 점검 화면

- 실적 등록/집계 전에 누락을 사전 목록으로 노출(문서 일정 경고 예방).
- 기준일 D 입력 → 미등록/누락 항목을 심각도별로 표시 → 각 항목에서 해당 마스터 화면으로 딥링크.

**심각도 분류 (원칙: 산출 불가=BLOCK, 품질 저하=WARN · 규칙별 기본표, 설정 조정 가능)**

| 심각도 | 규칙 |
|--------|------|
| BLOCK | CT/ST 미등록, 라우팅 누락, 라인/셀·설비 매핑 누락, 월력 미등록, 공정 마스터 부재 |
| WARN | 근무시간표 부분 누락, 무작업 사유코드 미분류, 모델 생성유형 공란 |

마감 미구현 단계에서는 라벨·집계 신뢰도 표시로만 사용하고, 향후 OEE 마감이 생기면 BLOCK만 마감 게이트에서 차단하고 WARN은 통과한다.

## 5. 딜리버리 계획 (수직 트레이서)

### Phase 0 — 발견/정비 (선행)

- CT/ST/SETUP·설비 위치(`IMCN_MACHINE`) 컬럼 실측, 기존 라우팅 버전 컬럼 유무 확인.
- `PLANTS`에 실제 라인/셀 레이아웃 데이터 정비 계획 수립, `IP_PRODUCT_LINE` ↔ `PLANTS` 라인 대응표 작성.
- 공통 버전 유틸(3.2) 뼈대 작성.

### Phase 1 — 트레이서: CT/ST/SETUP end-to-end

- 행-버전 스키마 생성 + 마이그레이션(기존 값 있으면 `VALID_FROM` 백필).
- as-of 조회 서비스 + 겹침 검증 적용.
- CT/ST 검증기 등록 + 준비도 화면 1탭.
- **완료 기준(goal)**: 임의 기준일로 CT/ST 유효행이 정확히 조회되고, 미등록 시 준비도 화면에 오류로 잡힌다(테스트 통과).

### Phase 2 — 패턴 복제(버전형 확장)

- 라우팅 → 월력 → 근무시간표 → 라인/셀·설비 매핑 순으로 동일 패턴 적용.
- `oee_resource` → PLANTS 참조 + 매핑으로 이관(기존 `oee/entry`·`oee/dashboard` 참조 리팩터 포함).

### Phase 3 — 단순형 보강 + 통합

- 공정·설비·모델·무작업사유코드에 마감검증 훅 + 공통 컬럼(수정자/수정일).
- 준비도 화면 전체 기준정보 통합.

### Phase 4 — 데이터 마이그레이션 마감

- 전 버전형 테이블 `VALID_FROM/TO` 백필, PLANTS 라인/셀 실데이터 확정, 매핑 초기 적재.

## 6. 리스크와 의존성

- **`oee_resource` 리팩터** — 기존 `oee/entry`·`oee/dashboard`가 참조 중. Phase 2에서 데이터 이관·참조 교체 필요(회귀 테스트 필수).
- **PLANTS 데이터 공백** — 조직트리에 실제 라인/셀이 채워져야 매핑이 성립. 데이터 정비가 개발과 나란히 진행되지 않으면 준비도 화면이 오류로만 가득 참(문서 일정 경고).
- **CT 실측 리드타임** — CT/ST 실측값 수집에 시간 소요. Phase 1은 스키마·로직 완성, 값 적재는 병행.
- **마감 부재** — 검증 API는 계약만 제공. 실제 잠금 효과는 향후 OEE 마감 구현 시 발생.

## 7. 협의 항목

### 7.1 확정됨 (2026-07-10 grill-me, §2 결정 9~12)

- ✅ CT/ST/SETUP 기존 데이터 — 산재 확인(`ROUTING_PROCESSES`·`PROCESS_MAPS`·`PROCESS_CAPAS`.`STD_TIME` 등) → 전용 마스터로 통합.
- ✅ `IP_PRODUCT_LINE` ↔ `PLANTS` — 공통키 매칭 + 시드승격.
- ✅ 후공정 셀 정의 — CELL 노드=집계단위, 설비매핑 선택.
- ✅ 준비도 심각도 — 산출불가=BLOCK, 품질저하=WARN 기본표.

### 7.2 남은 것 (결정 아님 · 데이터/현장 확인)

- 산재 `STD_TIME` 3곳 중 어느 값이 정본인지(이관 시 값 검증) — Phase 0 데이터 확인.
- PLANTS **후공정 셀 실분할**(현장이 셀을 어떻게 나누는지) — 현장 협의 필요.
- CT/ST **실측값 수집** 리드타임 — 스키마·로직과 병행, 값 적재는 후행 가능.
