## 1. 목적

본 문서는 은성 MES 고도화 중 수불관리, 재고실사, 창고마감, 단가/재고금액 평가, BOM/라우팅 기반 백플러시, OEE 관리 기능의 개발 방향을 정리하고 고객 합의를 얻기 위한 개발계획서 초안이다.

ERP 인터페이스는 이번 개발계획의 직접 구현 범위에서 제외하고, 수불/마감 기준 안정화 후 후속 연계 과제로 검토한다.

### 1.1 AS-IS / TO-BE / GAP 분석

본 문서는 분석보고서 구조를 따른다. 오늘의 현황(AS-IS), 개발 후의 모습(TO-BE), 그 격차(GAP)를 먼저 정의하고, 이후 절은 GAP 해소 순서로 전개한다.

| 영역 | AS-IS (오늘) | TO-BE (내일) | GAP (격차) | 해소 절 |
|---|---|---|---|---|
| 용어/기준 | "재공", "완제품대기재고" 등 혼용 | 창고+재고 결합 6종 표준 용어 통일 | 용어 표준 확정과 화면/코드 반영 | 3절 |
| 재고 원장 | 원자재창고까지만 원장, 공정 이후 추적 사각지대 | 6개 재고 구분 전 구간 원장, 바코드/UID 낱개 추적 | 신규 수불 테이블 9종, 매거진/포장 매핑 | 4~8절 |
| 재고금액 | 단가 테이블 존재하나 평가 미연결 | 총평균법 월마감 평가 (원자재 한정) | 평가 로직 + 단가 등록 운영 정착 | 9절 |
| 자재 소비 | 백플러시 없음, 소비 미기록 | 실적 등록 시 생성유형 3종 기준 자동 백플러시 | 생성유형 기준정보 + BOM/라우팅 정비 | 10절 |
| 실사/마감 | 실사/마감 프로세스 부재, 데이터 잠금 없음 | 조정 거래 실사 + 오류 해소형 마감 + 스냅샷 | 검증 규칙, 스냅샷 구조, 실사 주기 확정 | 13~14절 |
| 기준정보 | CT/ST, 라우팅, 월력 등 미등록/산재 | 15종 기준정보를 3원칙으로 관리 | 기준정보 수집·정비 (개발 선행) | 17절 |
| OEE | 지표 없음, 손실 원인 데이터 부재 | 공정 기본/라인·셀·설비 상세 OEE와 직행율, 마감 스냅샷 | 시간 기준정보, 사유코드, 입력 체계 신설 | 16절 |
| 가시성 | 현황 파악 화면 부재 | 모니터링 디자인 시안 + 재고 조회 8관점 + PC/PDA 입력 | 화면 개발 + 디자인 시안 선택 | 11절, 15절 |

## 2. 전체 범위

개발 범위는 다음 기능을 전체 대상으로 한다.

- 원자재 수불
- 공정창고 원자재 수불
- 공정창고 반제품 수불
- 반제품창고 반제품 수불
- 공정창고 완제품 수불
- 완제품창고 완제품 수불
- 재고실사
- 창고마감
- 구매단가 및 판매단가 활용
- 원자재 재고금액 및 출고금액 평가
- BOM/라우팅 기준 백플러시
- OEE 입력, 집계, 모니터링, 분석

## 3. 재고 구분 용어

문서와 화면에서는 아래 용어를 기준으로 통일한다.

| 구분 | 설명 |
|---|---|
| 원자재창고 원자재재고 | 원자재창고에 보관 중인 원자재 |
| 공정창고 원자재재고 | 원자재창고에서 출고되어 공정창고가 보유 중인 원자재 |
| 공정창고 반제품재고 | 공정에서 생산되어 후공정 또는 반제품창고 이동 전 보유 중인 반제품 |
| 반제품창고 반제품재고 | 별도 반제품창고에 보관 중인 반제품 |
| 공정창고 완제품재고 | 최종공정/포장 완료 후 완제품창고 입고 전 공정창고가 보유 중인 완제품 |
| 완제품창고 완제품재고 | 완제품창고 입고 처리 완료 후 보관 중인 완제품 |

`완제품재공`, `완제품대기재고` 등 혼동 가능한 용어는 사용하지 않고, `공정창고 완제품재고`로 표현한다.

## 4. 현행 기준 테이블

### 4.1 원자재

| 용도 | 테이블 |
|---|---|
| 원자재 바코드 마스터 | `IM_ITEM_RECEIPT_BARCODE` |
| 원자재 입고 | `IM_ITEM_RECEIPT` |
| 원자재 출고 | `IM_ITEM_ISSUE` |
| 원자재창고 현재고 | `IM_ITEM_INVENTORY` |
| 공정창고 원자재 입고 | `IM_ITEM_WORKSTAGE_RECEIPT` |
| 공정창고 원자재 출고 | `IM_ITEM_WORKSTAGE_ISSUE` |
| 공정창고 원자재 현재고 | `IM_ITEM_WORKSTAGE_INVENTORY` |

원자재는 입고 시 고유 바코드를 부여하며, 바코드는 유일하다. LOT 분할 입고/출고는 현행 운영 중이고, 분할 시 모 바코드 이력도 보존한다.

### 4.2 단가 및 판가

| 용도 | 테이블 |
|---|---|
| 구매단가 | `IM_ITEM_UNIT_PRICE` |
| 판매단가/판가 | `IS_PRODUCT_SALE_PRICE` |

판매단가는 재고평가에 사용하지 않고, 판매/분석용으로 분리한다.

### 4.3 완제품 포장

| 용도 | 테이블 |
|---|---|
| 완제품 포장 헤더 | `IP_PRODUCT_PACK_MASTER` |
| 포장 내 제품 바코드/UID 상세 | `IP_PRODUCT_PACK_SERIAL` |

`IP_PRODUCT_PACK_MASTER.PACK_BARCODE`를 포장 단위 헤더로 보고, `IP_PRODUCT_PACK_SERIAL.PACK_BARCODE`와 연결한다. `IP_PRODUCT_PACK_SERIAL.BARCODE`는 실제 제품 바코드 또는 내부 UID로 사용한다.

## 6. 원자재 수불 구조

원자재 출고는 소비가 아니라 공정창고 원자재 입고로 이어지는 이동 거래로 정의한다.

1. 원자재 입고: `IM_ITEM_RECEIPT`, `IM_ITEM_RECEIPT_BARCODE`
2. 원자재창고 재고: `IM_ITEM_INVENTORY`
3. 원자재 출고: `IM_ITEM_ISSUE`
4. 공정창고 원자재 입고: `IM_ITEM_WORKSTAGE_RECEIPT`
5. 공정창고 원자재 재고: `IM_ITEM_WORKSTAGE_INVENTORY`
6. 생산실적 등록 시 BOM/라우팅 기준 백플러시
7. 공정창고 원자재 출고: `IM_ITEM_WORKSTAGE_ISSUE`

`ID_ITEM.AUTO_RECEIPT_YN`, `AUTO_ISSUE_YN`, `AUTO_ISSUE_PLAN_YN`은 원자재창고 입출고 자동화 기준으로 보며, 공정창고 인수 자동확정 기준은 별도 조건으로 검토한다.

## 7. 반제품 수불 구조

반제품은 개별 바코드 유무와 관계없이 매거진 단위로 구성하여 이동 처리한다. 현장 작업은 매거진 스캔으로 수행하되, 수불 원장은 매거진에 매핑된 반제품 바코드 또는 내부 UID 단위로 생성한다.

반제품 이동 방식은 두 가지를 지원한다.

1. 전공정 공정창고 반제품재고 -> 후공정 공정창고 반제품재고
2. 전공정 공정창고 반제품재고 -> 반제품창고 반제품재고 -> 후공정 공정창고 반제품재고

전공정에서 후공정으로 밀어내기 이동 시 후공정 입고는 자동 확정한다. 후공정은 매거진 단위 전수 인수를 원칙으로 하며 부분 인수는 허용하지 않는다. 예외는 보류, 반송, 불량, 재작업 거래로 처리한다.

## 8. 완제품 수불 구조

완제품은 개별 제품 바코드가 있는 품목과 없는 품목이 혼재한다.

- 개별 제품 바코드가 있는 경우: 실제 제품 바코드를 수불 단위로 사용
- 개별 제품 바코드가 없는 경우: 포장 시 내부 UID를 생성해 수불 단위로 사용
- 작업자는 포장/박스/매거진 바코드를 스캔
- 시스템은 `IP_PRODUCT_PACK_SERIAL`에 매핑된 제품 바코드/UID 단위로 수불 원장을 생성

완제품 흐름은 다음과 같다.

1. 포장/매거진 구성
2. `IP_PRODUCT_PACK_MASTER` 생성
3. `IP_PRODUCT_PACK_SERIAL`에 제품 바코드/UID 매핑
4. 공정창고 완제품 입고: `IP_PROD_WORKSTAGE_RECEIPT`
5. 제품창고 담당자 입고 처리
6. 공정창고 완제품 출고: `IP_PROD_WORKSTAGE_ISSUE`
7. 완제품창고 입고: `IP_PRODUCT_FG_RECEIPT`
8. 완제품창고 현재고: `IP_PRODUCT_FG_INVENTORY`
9. 출하/반품/조정: `IP_PRODUCT_FG_ISSUE` 및 관련 조정 거래

완제품 입고는 제품창고 담당자가 포장/박스/매거진 번호를 스캔하여 처리한다. 입고는 포장 단위로 일괄 수행하지만, 수불 반영은 포장 내 제품 바코드/UID 단위로 수행한다.

## 9. 재고평가 및 출고금액

재고금액/출고금액 평가는 원자재 기준으로 제한한다.

대상:

- 원자재창고 원자재재고
- 공정창고 원자재재고
- 원자재창고 출고금액
- 공정창고 원자재 백플러시 출고금액

제외:

- 반제품 원가
- 완제품 원가
- 제품/반제품 재료비 전이
- 노무비, 제조간접비, 설비비

평가단가 기준:

- 기본: 총평균법
- 대안: 이동평균법
- 품목별 평가법 혼용은 허용하지 않음

총평균법은 월마감 시 `기초재고금액 + 당월입고금액`을 `기초재고수량 + 당월입고수량`으로 나누어 월평균단가를 산출하고, 해당 월 재고금액 및 출고금액 평가에 적용한다.

## 10. BOM/라우팅 및 백플러시

모델마스터에는 작업지시 생성유형을 관리한다.

| 코드 | 의미 |
|---|---|
| `BOM_LEVEL` | 모델/품번 BOM 레벨 기준 |
| `ROUTING` | 라우팅 공정 기준 |
| `MANUAL_HOLD` | 자동 생성/자동 백플러시 보류 |

생성유형이 비어 있는 경우 임시 기본값을 적용하지 않고 보류 처리한다. 잘못된 자재 소비가 발생하지 않도록 기준정보 오류로 식별한다.

백플러시는 작업지시 공정실적 등록 시점에 수행한다.

- `BOM_LEVEL`: 모델/품번 BOM 기준 소요량 사용
- `ROUTING`: 라우팅 공정 기준 소요량 사용
- `MANUAL_HOLD`: 자동 백플러시 금지, 보류 목록 처리

## 11. 수불부 조회

수불부 조회는 아래 재고 구분을 기준으로 제공한다.

- 원자재창고 원자재
- 공정창고 원자재
- 공정창고 반제품
- 반제품창고 반제품
- 공정창고 완제품
- 완제품창고 완제품

기준월별 수불부는 다음 항목을 제공한다.

- 기초
- 입고
- 출고
- 반품
- 불량/폐기
- 조정
- 현재고

기본 조회 조건:

- 기준월 또는 기간
- 창고구분
- 재고구분

상세 조회 조건:

- 품목/모델
- 바코드/UID
- 매거진/포장바코드
- 작업지시번호
- 공정
- 라인
- 설비
- 거래유형
- 거래상태
- 고객사
- 출하지시/출하번호

재고 조회는 단일 화면이 아니라 같은 수불 원장을 아래 8가지 관점으로 나누어 제공한다.

1. 창고 요약: 4개 창고 총수량/품목수/재고금액/구성비, 이동 중 재고(`TRANSFER_OUT` 후 미입고), 재고 이상 신호(음수재고, 단가 미등록, 장기 체류)
2. 재고현황: 창고/재고구분/품목/바코드·UID/포장·매거진 단위 현재고
3. 수불부: 기초-입고-출고-반품-불량/폐기-조정-현재고
4. 공정창고: 라인/공정이 보유 중인 원자재/반제품/완제품과 인수/백플러시 대기 상태
5. 바코드/LOT 추적: 모 바코드-분할-이동-백플러시 소비의 정방향/역방향 추적, 분할 계보
6. 재고 연령: 입고 후 경과일 구간(예: 30일/90일) 분석, 장기 체류 재고 식별 (실사 우선 순회/폐기 검토 대상)
7. 안전재고 과부족: 안전재고 대비 충족률, 최근 30일 백플러시 실적 기반 일평균 소비와 소진 예상일
8. OEE 이력: 마감 스냅샷 기준 지표 이력

어떤 관점에서 시작하든 거래 -> 바코드/UID -> 작업지시 -> 포장번호 -> 마감상태까지 원장으로 드릴다운할 수 있어야 한다. 안전재고 기준값의 출처(품목 기준정보 신설 여부)는 협의 항목이다.

## 13. 재고실사

재고실사는 아래 4개 창고 영역별로 제공한다.

- 원자재창고 실사
- 공정창고 실사
- 반제품창고 실사
- 완제품창고 실사

실사 원칙:

- 실사수량으로 현재고를 직접 덮어쓰지 않는다.
- 장부수량과 실사수량 차이를 계산한다.
- 차이수량이 양수이면 조정입고 거래를 생성한다.
- 차이수량이 음수이면 조정출고 거래를 생성한다.
- 모든 재고 차이는 수불 조정 거래로 반영한다.
- 실사 확정 후 수정은 제한하고, 필요 시 실사취소/재실사 이력으로 관리한다.

완제품 실사는 포장/박스/매거진 스캔으로 수행하되, 시스템 반영은 `IP_PRODUCT_PACK_SERIAL`의 제품 바코드/UID 단위로 처리한다.

실사 주기는 창고별로 협의하여 확정한다 (매월, 분기 순환실사, 연 1회 전수 등).

실사를 수행하지 못한 월에도 창고마감은 정상 진행한다. 마감의 전제 조건은 실사가 아니라 원장 정합성 검증이며, 마감 오류 해소 후 마감 스냅샷이 생성되면 차월 기초재고는 장부(원장) 기준으로 이월되어 운영이 계속된다. 이 경우 장부-실물 차이는 다음 실사까지 누적된 채 이월되므로, 재고 연령 분석과 이동 중 재고 경보를 실사 공백기의 보조 신호로 활용한다.

## 16. OEE 설계

OEE는 고객 제안서의 단순 지표 화면 수준을 넘어서, 라우팅/공정/라인/셀/설비/실적/무작업 사유를 연결한 분석 구조로 설계한다.

### 16.1 관리 기준

- OEE 기본 집계 단위는 공정이다.
- 상세 집계 단위는 라인/셀/설비이다.
- 공정 정보는 제안서 공정흐름 및 `IP_PRODUCT_WORKSTAGE`를 기준으로 한다.
- 설비 정보는 `IMCN_MACHINE`을 기준으로 한다.
- 라인 정보는 기존 `IP_PRODUCT_LINE` 테이블을 기준으로 하고, 셀 코드 체계는 신설 협의 대상이다.
- SMT 공정은 라인 단위 설비로 관리한다.
- 후공정은 셀 방식으로 관리한다.
- 설비가 명확한 공정은 설비 단위까지 집계하고, 설비 구분이 약한 공정은 라인/셀 또는 공정 단위로 집계한다.

### 16.2 라우팅 기준

라우팅은 거래처/주요품목별 공정 흐름표를 기준으로 구성한다. 기본 공정 후보는 다음과 같다.

- SMT
- 성능검사
- ICT
- 디스펜서
- WAVE
- 코팅
- 라우터
- 수납땜
- 외관검사
- 제품창고

거래처/품목별로 `a`, `b`, `c`와 같이 표시된 순서를 실제 라우팅 순서로 해석한다. 제품이 실제로 경유하지 않는 공정은 해당 제품의 OEE 분모에서 제외한다.

라우팅 기준 키:

- 거래처
- 모델/품목
- 라우팅 유형
- 공정 순번
- 공정 코드
- 라인/셀 코드
- 설비 코드
- 적용 시작일/종료일

### 16.3 시간 기준

OEE 산출을 위해 월력과 근무시간 관리표를 신설 또는 확장한다.

월력 관리 항목:

- 기준일
- 근무일/휴일/공휴일/특근일 구분
- 사업장/공장
- 적용 시작일/종료일

근무시간 관리 항목:

- 공정
- 라인/셀
- 근무조
- 근무 시작/종료 시간
- 휴식/식사 시간
- 계획정지 시간
- 특근/단축근무 예외
- 설비별 예외 시간

시간 개념은 아래와 같이 구분한다.

| 항목 | 정의 | 주요 용도 |
|---|---|---|
| CT | 실제 또는 표준 사이클 타임. 1개 생산에 필요한 기준 시간 | OEE 성능율 산출 |
| ST | 표준작업시간. 작업 표준, 공수, 계획 수립에 사용하는 시간 | 생산계획, 부하, 표준공수 |
| Takt Time | 고객 요구수량을 맞추기 위한 목표 생산 간격. `가용시간 / 필요수량` | 수요 대비 필요 속도 판단 |
| Neck Time | 라우팅 중 가장 시간이 긴 병목 공정 시간 | 병목공정 분석 |
| UPH/UPD | 시간당/일당 생산량 | 사후 분석 지표 |

OEE 성능율은 CT를 기준으로 계산한다.

```text
성능율 = 표준 CT * 생산수량 / 실제 가동시간
```

UPH/UPD는 시간이 지난 뒤 분석하는 결과 지표로 사용하고, 성능율의 직접 입력값으로 사용하지 않는다.

### 16.4 무작업/비가공 사유 체계

무작업 사유는 통제가능 무작업과 통제불가 무작업으로 구분한다. 사유코드는 초안을 먼저 제시하고 고객 검토 후 확정한다.

사유코드 기본 구조:

- 통제구분: 통제가능, 통제불가
- 손실유형: 무작업, 비가공, 성능손실, 품질손실
- 귀속지표: 가동율, 성능율, 양품율
- 공정/라인/셀/설비 적용 범위
- 사용 여부

초기 사유 분류 후보:

| 대분류 | 예시 | 통제구분 예시 | 귀속지표 |
|---|---|---|---|
| 계획/지시 | 생산계획 없음, 작업지시 대기 | 통제가능 | 가동율 |
| 자재/전공정 대기 | 자재 미입고, 전공정 대기, 피더/자재교체 지연 | 통제가능/통제불가 | 가동율/성능율 |
| 설비/치공구 | 설비고장, 치공구 이상, 순간정지 | 통제가능 | 가동율/성능율 |
| SET UP/교체 | 모델변경, 조건변경, 초기세팅 | 통제가능 | 가동율 |
| 작업자/인원 | 작업자 부재, 인원부족, 교육 | 통제가능 | 가동율 |
| 품질/검사 | 검사 재시도, 불량 조치, 고객사 불량 반입 | 통제가능/통제불가 | 성능율/양품율 |
| 공정/조건 | 속도저하, 표준속도 미달, 조건 안정화 | 통제가능 | 성능율 |
| 물류/이동 | 매거진 대기, 이동 대기, 창고 인수 대기 | 통제가능 | 가동율 |
| 시스템/스캔 | PDA 장애, 바코드 오류, 실적 등록 지연 | 통제가능 | 가동율 |
| 기타 | 고객 지정 사유 | 별도 지정 | 별도 지정 |

순간정지, 속도저하, 표준속도 미달 등은 작업자가 매번 수량화하기 어렵기 때문에 시스템 계산을 우선한다. 생산실적, CT, 실제 가동시간으로 이론 생산수량 대비 부족분을 계산하고, 원인 분석이 필요한 경우 작업자 또는 관리자가 사유를 보정 입력하도록 한다.

### 16.5 입력 원칙

입력된 값으로 시스템이 계산할 수 있는 항목은 자동 산출한다. 수작업 입력은 분석 가치가 높고 자동 산출이 어려운 항목에 한정한다.

자동 산출 대상:

- 계획시간
- 부하시간
- 가동시간
- 비가동시간
- 생산수량
- 검사수량
- 양품수량
- 불량수량
- 성능손실
- UPH/UPD
- 양품율
- 직행율

작업자/PDA/PC 입력 대상:

- 무작업 시작/종료
- 무작업 사유
- 통제가능/통제불가 구분
- 계획정지/비계획정지 구분
- 전공정 대기
- 자재 대기
- 설비 고장
- SET UP 지연
- 작업자 부재
- 검사 재시도 등 원인 보정

PDA 무작업 입력 표준 작업순서:

1. 로그인·라인 확인 (사번 스캔)
2. 설비 바코드 스캔
3. 무작업 시작 (시각 자동 기록)
4. 사유 선택 (버튼 1터치. 통제구분/손실유형/귀속지표는 사유코드에 내장되어 작업자가 따로 고르지 않는다)
5. (선택) 기타 사유 메모
6. 무작업 종료 (시각 자동 기록. 시간 중복/공백 검증은 마감 단계에서 수행)

PDA 사유 버튼은 사유 대분류 기준 4그룹 16종 초안으로 구성한다 (자재/전공정, 설비/치공구·SET UP, 작업자/계획, 품질/시스템/기타). 순간정지, 속도저하 등 시스템 자동 계산 항목은 버튼에 두지 않는다. 무작업 시작/종료 누락은 OEE 마감 오류로 잡혀 반드시 보정된다.

### 16.6 지표 정의

OEE 주요 지표:

- 가동율 = 실제 가동시간 / 부하시간
- 성능율 = 표준 CT 기준 생산 가능시간 / 실제 가동시간
- 양품율 = 양품수량 / 생산수량
- OEE = 가동율 * 성능율 * 양품율

직행율은 양품율과 별도로 관리한다.

- 공정 직행율 = 재작업/재검 없이 해당 공정을 1회 통과한 수량 / 해당 공정 투입수량
- 라우팅 직행율 = 전체 라우팅을 재작업/재검 없이 통과한 수량 / 최초 투입수량

양품율은 최종 양품 비율이고, 직행율은 재작업/재검 없이 한 번에 통과했는지를 보는 흐름 품질 지표이다.

### 16.7 실시간 집계와 마감 스냅샷

OEE는 실시간 조회와 마감 스냅샷을 구분한다.

- 실시간 조회: 현재 생산실적, 작업지시, 무작업 입력, 설비/라인 상태를 기준으로 임시 계산한다.
- 마감 스냅샷: 근무조 또는 일자 마감 시 확정 데이터를 저장하여 과거 조회가 변하지 않도록 한다.

스냅샷 집계 축:

- 기준일
- 근무조
- 모델/품목
- 작업지시
- 라우팅
- 공정
- 라인/셀
- 설비
- 작업자/작업조

### 16.8 OEE 마감

OEE 마감도 재고마감과 동일하게 오류 해소형 프로세스로 설계한다.

1. OEE 마감 실행
2. 사전 검증
3. 오류 목록 자동 생성
4. 담당자 조치
5. 조치 완료 확인
6. OEE 재마감
7. 성공 시 OEE 스냅샷 생성
8. 마감 후 기준일 데이터 수정 제한
9. 필요 시 마감취소/재마감 이력 관리

OEE 마감 오류 예시:

- 생산실적 미확정
- 무작업 시작/종료 누락
- 시간 중복/공백
- CT/ST 미등록
- 라우팅 누락
- 라인/셀/설비 매핑 누락
- 작업지시 연결 누락
- 양품/불량 수량 불일치
- 직행율 산출 불가
- 월력/근무시간 미등록

## 17. 기준정보 관리

수불과 OEE의 마감 오류 중 상당수(단가 미등록, CT/ST 미등록, 라우팅 누락, 라인/셀/설비 매핑 누락, 월력/근무시간 미등록)는 기준정보 문제다. 기준정보는 한 번 등록하고 끝나는 값이 아니라, 사용처/관리 형식/관리 주기/관리 주체/오류 시 영향이 정의된 상태로 관리한다.

### 17.3 관리 3원칙

1. 기간 버전 관리: 기준정보는 값을 덮어쓰지 않고 적용 시작일/종료일을 가진 기간 버전으로 관리한다. 단가/CT/라우팅이 바뀌어도 과거 마감 스냅샷과 과거 OEE를 당시 기준으로 재현할 수 있어야 한다.
2. 마감 검증 연동: 기준정보 누락은 화면 경고로 끝내지 않고 재고마감/OEE 마감의 오류 목록으로 잡는다. 생성유형 공란 -> 백플러시 보류와 같은 원칙으로, 기본값으로 조용히 넘어가지 않는다.
3. 변경 이력과 승인: 누가/언제/무엇을/왜 바꿨는지 이력을 남기고, 영향이 큰 항목(단가, CT/ST, 라우팅, BOM)은 승인 후 적용한다. 지표가 움직인 원인이 현장인지 기준정보 변경인지 구분되어야 한다.

모든 기준정보 화면은 적용 시작/종료일, 상태(승인/승인 대기/이력/미등록), 수정자/수정일 컬럼을 공통으로 갖는다. 과거 버전은 삭제하지 않고 이력 상태로 보존한다.

### 17.4 정비 일정

기준정보 정비(특히 CT 실측)는 수집 기간이 필요하므로 개발보다 먼저 시작한다. CT/ST 기존 데이터 존재 여부, 라인/셀/설비 매핑 상세, 안전재고 기준값 출처는 남은 협의 항목이다. 개발 완료 시점에 기준정보가 비어 있으면 마감 오류 목록만 가득한 시스템이 되므로, 기준정보 수집 계획을 개발 일정과 나란히 확정한다.

| 공정은 제안서 페이지와 기존 `IP_PRODUCT_WORKSTAGE` 등록 기준으로 설정한다. | OEE와 라우팅 기준에서 `IP_PRODUCT_WORKSTAGE`를 공정 기준으로 명시하였다. |
| 설비는 `IMCN_MACHINE` 테이블을 이용할 예정이다. | OEE 상세 집계 기준에서 `IMCN_MACHINE`을 설비 기준으로 명시하였다. |
| ERP 인터페이스는 보류한다. | 구현 범위에서 ERP 인터페이스를 제외하고 후속 과제로 명시하였다. |
| 범위는 전체로 잡아야 한다. | 원자재, 공정창고, 반제품창고, 완제품창고, 실사, 마감, 수불부, OEE 전체를 범위에 포함하였다. |

### 20.2 원자재 바코드와 기존 원자재 수불

| 대화 내용 | 문서 반영 |
|---|---|
| 모든 자재는 입고 시 고유 바코드를 이미 관리하고 있다. | 원자재는 바코드 유일성을 전제로 하였다. |
| 바코드 자체로 유일하다. | 원자재 수불의 기본 식별자를 바코드로 보았다. |
| 현재 이미 롯트 분할 입고/출고는 정상 처리 중이다. | 기존 운영 기능으로 인정하고 신규 설계의 전제에 포함하였다. |
| 분할 시 모 바코드는 남기고 있다. | `IM_ITEM_RECEIPT_BARCODE` 설명에 모 바코드 보존 전제를 반영하였다. |
| `IM_ITEM_RECEIPT_BARCODE`가 마스터 테이블이다. | 현행 기준 테이블의 원자재 바코드 마스터로 명시하였다. |
| `IM_ITEM_RECEIPT`는 입고이다. | 현행 원자재 입고 테이블로 명시하였다. |
| `IM_ITEM_ISSUE`는 출고이다. | 원자재창고 출고 테이블로 명시하였다. |
| `IM_ITEM_INVENTORY`는 원자재 재고 정보이다. | 원자재창고 현재고 테이블로 명시하였다. |
| 공정보유재고로 관리해야 한다. | 원자재창고 출고 후 공정창고 원자재재고로 관리하는 구조를 정의하였다. |

### 20.3 백플러시와 자동 처리 기준

| 대화 내용 | 문서 반영 |
|---|---|
| 공정에서 작업지시가 내려가고 실적을 잡는 공정에서 BOM 기준 또는 라우팅 기준으로 백플러시가 일어나면 된다. | 생산실적 등록 시 BOM/라우팅 기준으로 공정창고 원자재를 백플러시 출고하는 구조를 반영하였다. |
| 모델마스터 `IP_PRODUCT_MODEL`에 해당 모델의 작업지시 생성유형을 관리한다. | BOM 레벨 제품인지 라우팅 방식 제품인지 구분하는 하이브리드 기준으로 반영하였다. |
| 추천 방식으로 진행한다. | 하이브리드 방식을 기본 방향으로 채택하였다. |
| 원자재출고가 공정입고가 되고, 백플러시에 의해 공정출고가 되는 수불구조를 원한다. | `IM_ITEM_ISSUE` -> `IM_ITEM_WORKSTAGE_RECEIPT` -> `IM_ITEM_WORKSTAGE_ISSUE` 흐름으로 반영하였다. |
| 현재 공정 수불 테이블은 `IM_ITEM_WORKSTAGE_RECEIPT`, `IM_ITEM_WORKSTAGE_ISSUE`, `IM_ITEM_WORKSTAGE_INVENTORY`가 있다. | 원자재 공정창고 수불의 기존 테이블로 반영하였다. |
| 옵션으로 처리하면 자동으로도 되고 제어도 된다는 판단이 있었다. | 자동 처리 기준은 옵션화하되 기존 자동입출고 의미와 분리해야 한다고 정리하였다. |
| 자동 인수 설정을 어디에 둔다는 것인지 질문이 있었다. | 설정 위치가 공정별이 아니라 자재품목별이라는 후속 정정 내용을 반영하였다. |
| 공정별 기준이 아니고 자재품목별로 설정되어야 한다. | 자재 품목 기준 자동 처리 조건으로 방향을 보정하였다. |
| 이미 `auto_issue_yn`이 있을 수 있다. | 기존 `AUTO_ISSUE_YN` 존재 가능성을 반영하였다. |
| `AUTO_RECEIPT_YN`은 원자재창고 전표 입고 기준이고, `AUTO_ISSUE_YN`도 원자재창고 출고 자동 기준이다. | 기존 필드는 원자재창고 자동 입출고 기준으로만 해석한다고 명시하였다. |
| 공정에서 자동으로 인수한다는 개념과는 다르므로 별도 조건이 필요하다. | 공정창고 인수 자동확정 기준은 별도 조건으로 검토한다고 명시하였다. |

### 20.4 단가, 판가, 재고평가

| 대화 내용 | 문서 반영 |
|---|---|
| 구매단가는 이미 테이블과 화면이 레거시로 구현되어 있다. | 기존 단가 테이블 활용을 전제로 하였다. |
| 구매단가 테이블은 `IM_ITEM_UNIT_PRICE`이다. | 현행 기준 테이블에 반영하였다. |
| 판가는 `IS_PRODUCT_SALE_PRICE`이다. | 현행 기준 테이블에 반영하였다. |
| 재고평가를 위한 단가는 이동평균으로 하자는 고객 요구가 있었다. | 재고평가 방식 후보에 이동평균법을 포함하였다. |
| 옵션을 설정해서 재고평가단가를 선택할 수 있게 하는 의견이 있었다. | 방식 선택 가능성은 검토하되 품목별 혼용은 금지로 정리하였다. |
| 단가를 다르게 평가하면 안 되고 한 개로 정해야 한다. | 동일 기간에는 하나의 평가방식을 적용해야 한다고 명시하였다. |
| 이동평균법과 총평균법은 선택 가능하게 한다. | 총평균법 기본, 이동평균법 대안으로 정리하였다. |
| 일단 총평균법으로 잡는다. | 본 단계 기본값을 총평균법으로 명시하였다. |
| 원가관리는 필요 없고 재료비만 관리하면 된다. | 노무비, 제조경비, 설비비 배부를 제외하였다. |
| 제품재료비 평가는 필요 없다. | 완제품/반제품 재료비 평가는 제외하였다. |
| 창고재고 금액평가와 출고금액 평가만 하면 된다. | 원자재창고/공정창고 원자재 재고금액 및 원자재 출고금액 평가로 범위를 제한하였다. |

### 20.5 완제품 수불

| 대화 내용 | 문서 반영 |
|---|---|
| 제품재고도 해야 한다. | 완제품창고 완제품재고를 범위에 포함하였다. |
| 완제품은 개별 바코드가 있는 것이 있고 없는 것이 있다. | 완제품 바코드 유무 혼재를 전제로 하였다. |
| 개별 바코드가 없는 것은 박스에 담아서 라벨을 붙여 관리한다. | 박스/매거진 라벨과 내부 UID 방식을 반영하였다. |
| 창고 담당자가 입고 처리를 진행한다. | 완제품창고 입고는 생산 자동입고가 아니라 창고 담당자 입고 처리로 명시하였다. |
| 입고 전 상태는 공정창고에 있는 완제품재고이다. | 용어를 공정창고 완제품재고로 통일하였다. |
| 입고를 잡을 때 포장해서 매거진 라벨을 발행하고 매거진/박스번호를 스캔하여 일괄 입고한다. | `IP_PRODUCT_PACK_MASTER`와 `IP_PRODUCT_PACK_SERIAL` 기준 포장 단위 입고를 반영하였다. |
| 완제품 입고는 분할 입고가 없다. | 완제품창고 입고는 포장 단위 일괄 입고로 정리하였다. |
| 일부 수량만 출고되는 경우가 있다. | 출고는 박스 스캔 입력 후 상세 제품 단위 일부 출고가 가능해야 한다고 반영하였다. |
| 출하박스에서 일부 수량만 반품되는 경우도 있다. | 출하반품은 포장 상세 제품 단위로 반영해야 한다고 정리하였다. |
| 박스를 스캔하더라도 그 안에 매핑된 개별 제품으로 수불을 일으켜야 한다. | 완제품 수불 원장은 `PACK_BARCODE + 제품 바코드/UID` 상세 단위로 생성한다고 명시하였다. |
| `IP_PRODUCT_PACK_MASTER`는 완제품 포장 데이터이다. | 완제품 포장 헤더로 반영하였다. |
| `IP_PRODUCT_PACK_SERIAL`은 완제품 포장된 제품바코드 모음이다. | 포장 상세 제품 바코드/UID 목록으로 반영하였다. |
| `IP_PRODUCT_FG_RECEIPT`, `IP_PRODUCT_FG_ISSUE`, `IP_PRODUCT_FG_INVENTORY`는 완제품 창고 수불 데이터이다. | 완제품창고 수불 기존 테이블로 반영하였다. |
| 개별 바코드가 없는 완제품은 실물과 차이가 발생할 수 있지만 내부 UID 방식을 시도한다. | 내부 UID 파일럿 검증 항목으로 반영하였다. |

### 20.6 공정수불, 반제품, 완제품 공정창고

| 대화 내용 | 문서 반영 |
|---|---|
| 공정수불에는 원자재뿐 아니라 반제품과 완제품이 모두 있다. | 공정창고를 원자재, 반제품, 완제품으로 분리하였다. |
| `IM_`으로 시작하는 테이블은 원자재 수불데이터이다. | 반제품/완제품 공정수불은 `IM_` 계열과 분리해야 한다고 명시하였다. |
| 반제품 공정수불은 `IP_ASSY_WORKSTAGE_RECEIPT`, `ISSUE`, `INVENTORY`가 필요하다. | 신규/보완 대상 테이블에 반영하였다. |
| 완제품 공정수불은 `IP_PROD_WORKSTAGE_RECEIPT`, `ISSUE`, `INVENTORY`가 필요하다. | 신규/보완 대상 테이블에 반영하였다. |
| 해당 완제품 공정수불 테이블은 지금 없고 새로 생성해야 한다. | 신규 생성 대상임을 명시하였다. |
| 반제품도 바코드가 있는 것이 있고 없는 것이 있다. | 반제품 바코드 유무 혼재를 전제로 하였다. |
| 반제품은 바코드 유무와 관계없이 매거진 단위로 구성해서 이동처리한다. | 반제품 매거진 구성 및 내부 UID 필요성을 반영하였다. |
| 후공정은 무조건 전수 인수이다. | 후공정 자동 전수 인수로 반영하였다. |
| 전공정에서 후공정으로 밀어내기 방식으로 수불한다. | 전공정 출고와 후공정 자동입고의 push 흐름을 반영하였다. |
| 후공정 인수는 당연히 자동이다. | 후공정 자동입고 전제로 반영하였다. |
| 매거진 구성을 하면 매거진 마스터는 별도로 생성되고 있다. | 매거진/포장 마스터와 상세 매핑을 수불 기준정보로 활용하는 방향을 반영하였다. |

### 20.7 수불부, 실사, 마감

| 대화 내용 | 문서 반영 |
|---|---|
| 수불원장 조회 방식에 대해서도 논의가 남아 있다. | 수불부 조회 절과 남은 협의 항목에 반영하였다. |
| 공정 수불부도 조회되어야 한다. | 원자재 수불부, 공정 수불부, 반제품창고 수불부, 완제품 수불부를 화면 후보에 포함하였다. |
| 세 군데 모두 재고마감 테이블과 재고실사 테이블이 있어야 한다. | 원자재창고, 공정창고, 완제품창고뿐 아니라 반제품창고까지 포함해 4개 영역으로 확장하였다. |
| 원자재창고마감, 공정창고마감, 완제품창고마감 및 모든 실사가 있어야 한다. | 4개 창고 마감 및 실사로 정리하였다. |
| 용어를 재고/재공으로 혼용하고 있었다. | 완제품재공 대신 공정창고 완제품재고로 용어를 정리하였다. |
| 공정창고 완제품재고로 하면 된다. | 용어 표준에 반영하였다. |
| 반제품창고, 반제품재고도 있어야 한다. | 반제품창고 반제품재고를 추가하였다. |
| 재고는 모두 수불처리로 조정한다. | 실사 차이는 현재고 덮어쓰기가 아니라 조정입고/조정출고 거래로 반영한다고 명시하였다. |
| 일은 항상 마무리가 되어야 하며 제한하고 끝나면 안 된다. | 마감은 오류 해소형 프로세스로 설계한다고 명시하였다. |

### 20.8 OEE 라우팅과 관리 단위

| 대화 내용 | 문서 반영 |
|---|---|
| 상단 SMT, 성능검사, ICT, 디스펜서, WAVE, 코팅, 라우터, 수납땜, 외관검사가 공정흐름이다. | OEE 라우팅 기준 공정 후보로 반영하였다. |
| 거래처별로 공정을 타는 순서가 `a`, `b`, `c`로 표시되어 움직인다. | `a`, `b`, `c`를 라우팅 순서의 근거로 명시하였다. |
| 이 흐름이 라우팅정보의 근간이다. | 라우팅 매트릭스를 기준정보 근거로 정의하였다. |
| 고객의 OEE 제시안보다 시스템은 더 깊이 있게 제안해야 한다. | OEE를 라우팅/공정/라인/셀/설비/시간/사유 연계 구조로 확장하였다. |
| OEE 관련 공정단위, 설비단위로 무작업 또는 비가공 원인을 등록한다. | OEE 관리 기준과 입력 원칙에 반영하였다. |
| PDA 또는 PC를 통해 무작업 원인을 등록한다. | PC/PDA 입력 화면 목업과 입력 기준에 반영하였다. |
| SMT 공정은 라인단위 설비로 관리한다. | OEE 관리 기준에 SMT 라인 단위 설비 관리로 반영하였다. |
| 후공정은 셀 방식으로 관리한다. | OEE 관리 기준에 후공정 셀 단위 관리로 반영하였다. |

### 20.9 OEE 사유코드와 지표

| 대화 내용 | 문서 반영 |
|---|---|
| 사유코드는 더 많이 있으며 먼저 제시하고 나중에 고객에게 다시 받을 것이다. | 초기 사유 분류 후보를 제시하고 고객 확정 항목으로 남겼다. |
| 무작업 사유는 통제불가 무작업과 통제가능 무작업으로 세분화해야 한다. | 통제구분을 통제가능/통제불가로 설계하였다. |
| 직행율도 있어야 한다. | 공정 직행율과 라우팅 직행율을 정의하였다. |
| 월력 및 근무시간 관리표가 있어야 한다. | 월력 관리와 근무시간 관리 항목을 추가하였다. |
| 비가공/성능손실 입력에서 순간정지, 속도저하, 표준속도 미달, 피더/자재교체 지연, 검사 재시도 등을 어떻게 수치화할지 질문하였다. | 자동 계산 우선, 수작업 보정 입력 원칙으로 정리하였다. |
| 입력된 값으로 얻을 수 있는 것은 최대한 얻고, 수작업 입력을 하면 좋은 분석이 가능한 데이터는 입력을 유도한다. | 자동 산출 대상과 작업자/PDA/PC 입력 대상을 분리하였다. |
| 성능율은 CT로 계산한다. | 성능율 산식을 CT 기준으로 명시하였다. |
| UPH는 시간이 지난 후 분석하는 것 아니냐는 질문이 있었다. | UPH/UPD는 사후 분석 지표로 명시하였다. |
| CT, ST, Tact Time, Neck Time 차이를 설명해야 한다. | 시간 개념 표에 CT, ST, Takt Time, Neck Time을 구분하였다. |
| Takt Time이 잘 이해되지 않는다는 질의가 있었다. | Takt Time은 고객 요구수량을 맞추기 위한 목표 생산 간격이라고 설명하였다. |
| 공정에서 가장 시간이 많이 걸리는 곳을 텍타임으로 알고 있었다. | 그것은 Takt Time이 아니라 Neck Time/Bottleneck Time이라고 구분하였다. |
| 왜 ST가 아니라 CT로 계산하는지 질의가 있었다. | 성능율은 실제 가동시간 대비 기준 속도를 보는 지표이므로 CT가 적합하다고 설명하였다. |
| 다른 시간들도 전부 관리해야 한다. | CT, ST, SET UP 표준시간, 모델변경 표준시간, 근무/휴식 시간을 관리 대상으로 반영하였다. |
