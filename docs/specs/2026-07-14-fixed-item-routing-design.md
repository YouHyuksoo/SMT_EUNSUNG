# 품목별 고정 생산 라우팅 설계

- 작성일: 2026-07-14
- 상태: 설계 승인됨 (구현 대기)
- 선행 설계: `docs/specs/2026-07-12-ip-routing-design.md`

## 1. 배경과 문제

선행 설계는 한 품목에 복수 라우팅 그룹과 하나의 활성 버전을 허용했다. 은성전장은
운영 복잡도를 낮추기 위해 HANES의 품목 기반 흐름을 참조하되, 품목마다 고정 라우팅
하나만 관리한다.

품질검사·품질조건·자주검사·회로 연결은 제외한다. 라벨은 종류를 선택하지 않고
공정별 발행 여부만 관리한다.

## 2. 목표 / 비목표

### 목표

- 현재 유효한 BOM이 있는 반제품(W)·완제품(F)에 고정 라우팅을 하나만 등록한다.
- 라우팅 코드는 품목코드와 동일하게 고정한다.
- 공정 순서, 실행유형, 작업지시 여부, 시간, 라벨발행 여부를 관리한다.
- BOM 하위품목을 라우팅의 한 공정에 배정한다.
- BOM 변경으로 생긴 불일치를 자동 삭제하지 않고 표시한다.

### 비목표

- 라우팅 버전·대안, 원자재(R) 라우팅
- 품질검사·품질조건·자주검사
- 회로 연결과 묶음 추적 라벨
- `NONE/BUNDLE/SG/FG` 라벨 종류 선택
- 실제 라벨 채번·인쇄
- 작업지시·생산실적 모듈 구현
- 레거시 `IP_PRODUCT_ROUTING_*` 정리

## 3. 설계 결정 요약

| 결정 | 선택 | 근거 |
|---|---|---|
| 라우팅 구조 | 품목별 단일 고정 라우팅 | 버전·활성 선택 없이 운영을 단순화한다. |
| 라우팅 코드 | `ITEM_CODE`와 동일 | 잘못된 매핑을 방지한다. |
| 저장 구조 | 기존 `IP_ROUTING_*` 3개 테이블 유지 | 기존 API·화면·무결성 구조를 재사용한다. |
| 라벨 설정 | 공정별 `LABEL_ISSUE_YN` | 여러 공정을 독립적으로 발행 대상으로 지정한다. |
| 라벨 종류 | 품목 구분 W/F로 후속 기능이 판단 | 잘못된 SG/FG 선택을 방지한다. |
| 자재 배정 | 현재 BOM 자재를 한 공정에 배정 | BOM은 유지하고 투입 공정만 관리한다. |
| BOM 변경 | 기존 배정 보존 및 불일치 표시 | 기준정보가 조용히 유실되지 않게 한다. |

## 4. 상세 설계

### 4.1 라우팅 그룹

- 등록 대상은 현재 유효한 `ID_ENG_BOM`의 상위품목인 W/F 품목이다.
- BOM이 없는 W/F는 `BOM 미등록`으로 표시하고 생성을 차단한다.
- 서버가 `ROUTING_CODE = ITEM_CODE`를 강제한다.
- 한 조직의 한 품목은 활성 여부와 무관하게 라우팅 그룹을 하나만 가진다.
- `USE_YN`은 고정 라우팅의 사용 여부이며 버전을 구분하지 않는다.
- DB는 `ORGANIZATION_ID + ITEM_CODE` 유니크 제약으로 중복을 차단한다.

기존의 활성 행만 제한하는 함수 기반 유니크 인덱스는 품목 전체를 제한하는 유니크
제약으로 대체한다.

기존 Oracle 환경에는 2026-07-14 증분 DDL로 컬럼·check·품목 유니크 제약을
적용한다. 신규 환경도 최종 구조로 생성되도록 `2026-07-12_ip_routing_tables.sql`의
생성 정의와 정확 검증 항목을 함께 갱신한다. 기존 함수 기반 인덱스와 중복 품목을
사전 검사한 뒤에만 새 유니크 제약으로 교체한다.

### 4.2 라우팅 공정

기존 공정 속성에 라벨 발행 여부를 추가한다.

| 속성 | 규칙 |
|---|---|
| `PROCESS_SEQ` | 라우팅 내 양의 고유 순번 |
| `WORKSTAGE_CODE` | 공정 마스터에서 선택 |
| `EXECUTION_TYPE` | `INTERNAL` 또는 `SUBCON` |
| `JOB_ORDER_YN` | 공정별 작업지시 생성 여부 |
| `SUBCON_SUPPLIER_CODE` | 외주일 때 필수, 내작일 때 `NULL` |
| `STANDARD_TIME`, `SETUP_TIME` | 0 이상 또는 `NULL` |
| `LABEL_ISSUE_YN` | `Y/N`, 기본값 `N`, 여러 공정에 `Y` 허용 |
| `USE_YN` | `Y/N` |

Oracle에는 다음 컬럼과 Y/N check constraint를 추가한다.

```sql
LABEL_ISSUE_YN VARCHAR2(1) DEFAULT 'N' NOT NULL
```

엔티티·DTO·프론트 타입·폼의 계약명은 `labelIssueYn`으로 통일한다.

### 4.3 공정별 자재

- 후보는 라우팅 품목의 현재 유효 BOM 하위품목이다.
- 한 자재는 한 라우팅 안의 한 공정에만 배정한다.
- `ALLOC_QTY`는 BOM 수량을 기본값으로 사용하고 양수 범위에서 조정할 수 있다.
- `ISSUE_METHOD`는 `BACKFLUSH` 또는 `PRE_ISSUE`다.
- BOM에서 빠진 기존 배정은 자동 삭제하지 않고 `BOM 불일치`로 표시한다.
- 불일치가 남은 라우팅은 후속 작업지시 생성 시 사용할 수 없다. 이번 범위에서는
  불일치 상태 제공까지만 구현한다.

### 4.4 화면과 API

- 기존 `PartSelect`는 100건만 조회하고 W/F·BOM 상태 계약이 없으므로 재사용하지 않는다.
- 라우팅 전용 후보 API는 W/F 품목을 서버 검색·페이지 처리하고 `bomRegisteredYn`,
  `selectableYn`을 반환한다. BOM이 없는 품목은 `BOM 미등록`으로 표시하되 선택할 수 없다.
- 제안 경로는 `GET /master/routing-groups/candidate-items`이며 검색어·페이지·건수를
  받는다. 정적 경로가 `:code`보다 먼저 매칭되도록 컨트롤러 순서와 회귀 테스트를 둔다.
- 라우팅 코드는 품목 선택 시 자동 설정하고 수정할 수 없게 한다.
- 공정 폼에 `라벨발행` Y/N 입력을 추가한다.
- 여러 공정에 발행 `Y`를 설정할 수 있다.
- 품질·자주검사 탭, 회로 선택, 라벨 종류 선택은 제공하지 않는다.
- 기존 `master/routing-groups` 및 공정별 자재 API 경로를 유지한다.

### 4.5 적용 범위

이번 구현은 라우팅 기준정보에 `LABEL_ISSUE_YN`을 저장·조회하는 것까지다.
향후 생산실적 기능은 완료 공정의 값을 읽어 라벨을 채번·인쇄하며, 라벨 구분은
라우팅 품목의 W/F 구분으로 결정한다.

## 5. 에러 처리

- W/F가 아니거나 현재 BOM이 없는 품목 생성: HTTP 422
- 라우팅 코드와 품목코드 불일치 또는 동일 품목 중복: HTTP 409
- 공정 순번·동시 수정 무결성 충돌: HTTP 409
- 외주처 누락, 음수 시간, Y/N 외 라벨 값, BOM 비소속 자재, 비양수 수량: HTTP 422
- 다른 공정에 이미 배정된 자재: 현재 배정 공정과 함께 HTTP 409
- 자식 공정·자재가 있는 상위 행 삭제: HTTP 409

## 6. 테스트 전략

### 구조·단위 테스트

- 엔티티·DTO·프론트 타입에 `labelIssueYn`이 있고 `issueLabelType`은 없는지 검증한다.
- 품질·자주검사·회로 필드와 UI가 없는지 검증한다.
- `ROUTING_CODE = ITEM_CODE`, 조직별 품목 단일성, W/F + 현재 BOM 제약을 검증한다.
- 후보 API의 W/F 필터, BOM 상태, 서버 검색·페이지 처리와 100건 초과 조회를 검증한다.
- 여러 공정에 `labelIssueYn='Y'`를 저장하고 재조회할 수 있는지 검증한다.
- BOM 중복 배정, 불일치 보존, 명시 삭제, 트랜잭션 롤백을 검증한다.
- 7/12 DDL 구조 테스트를 새 컬럼·품목 유니크 제약 기준으로 갱신하고, 7/14 증분 DDL의
  기존 구조 업그레이드·재실행·불일치 중단을 검증한다.

### 정적 검증

- `pnpm --filter @eunsung/backend test`
- `pnpm --filter @eunsung/backend exec tsc --noEmit --pretty false`
- `pnpm --filter @eunsung/frontend test`
- `pnpm --filter @eunsung/frontend typecheck`

### Oracle·런타임 검증

- DDL 전 실제 컬럼·제약·인덱스와 현재 건수를 기록한다.
- 기존 함수 기반 활성 인덱스와 조직·품목 중복 여부를 확인한 뒤 증분 DDL을 적용한다.
- `LABEL_ISSUE_YN`의 기본값·NOT NULL·check와 품목 유니크 제약을 확인한다.
- 식별 가능한 W/F BOM 품목으로 라우팅, 여러 발행 공정, 자재 배정을 생성·재조회한다.
- 검증 데이터는 자재 → 공정 → 그룹 순으로 정리하고 사전·사후 건수를 기록한다.
- 사용자가 3003/3100 서버를 기동한 경우 인증 백엔드 API → 프론트 프록시 →
  렌더된 화면 순으로 확인한다.

## 7. 제외한 대안

### 마지막 공정 자동 발행

저장 필드는 필요 없지만 여러 공정에서 발행할 수 있다는 업무 규칙을 표현하지 못한다.

### HANES 라벨 종류 그대로 이식

`NONE/BUNDLE/SG/FG`는 은성에서 제외한 묶음·회로 개념과 잘못된 W/F 선택 가능성을
들여오므로 제외한다.

### 품목 마스터에 라우팅 컬럼 추가

공정 순서와 자재 배정은 복수 행이므로 단일 컬럼으로 표현할 수 없고 품목 마스터의
책임만 불필요하게 키운다.

### 기존 복수 버전 설계 유지

유연성은 높지만 은성의 품목별 고정 운영 기준보다 복잡하고 라우팅 코드·활성 버전을
잘못 선택할 여지가 있어 제외한다.
