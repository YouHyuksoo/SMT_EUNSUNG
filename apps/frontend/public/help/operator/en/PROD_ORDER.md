# Work Order Management — Operator Guide

## System Purpose & Role
This screen issues and manages production orders for finished goods and semi-products — the **starting point for production results, inspection, shipping, and inventory**. Each work order has natural key `ORDER_NO` as PK, references item (`ITEM_CODE`) and routing (`ROUTING_CODE`), and uses self-reference (`PARENT_ID`) to form finished-good-to-semi-product hierarchy. Status transitions are only possible via dedicated APIs (start/hold/hold-release/complete/cancel); direct changes are blocked.

> API base path: `/production/job-orders`. Menu code `PROD_ORDER`, route `/production/order`.

## Data Structure
```
ITEM_MASTERS(item) ──ITEM_CODE──▶ JOB_ORDERS ◀──PARENT_ID── (semi-product child work orders)
        │                            │  │
        │                            │  └──ROOT_ORDER_NO── co-generation group root
ROUTING_GROUPS ──ROUTING_CODE────────┘  │
PROD_PLANS ──PLAN_NO────────────────────┤
                                        │
                            PROD_RESULTS(results) ◀──ORDER_NO── (result aggregation: GOOD_QTY/DEFECT_QTY)
                            FG_LABELS(finished goods barcode) ◀──ORDER_NO── (PRE_ISSUE pre-issuance)
```
- BOM expansion: From `BOM_MASTERS`(parentItemCode → childItemCode), only `SEMI_PRODUCT` items selected for recursive child work order creation.
- Result aggregation: On list/completion, sum of PROD_RESULTS good/defect (excluding CANCELED) is reflected in `GOOD_QTY`/`DEFECT_QTY`.

> Note (code inconsistency): The grid's right-side `sqlQuery` preview shows `PROD_ORDERS`, but **the actual entity/table name is `JOB_ORDERS`** (entity `@Entity({ name: 'JOB_ORDERS' })`). Operational basis is `JOB_ORDERS`.

## ① Work Order — JOB_ORDERS (All Columns)

| Screen Field | DB Column | Role / Meaning · Operational Notes |
|------|------|------|
| Work Order No. | `ORDER_NO` | PK (natural key, varchar2 50). Auto-generated by server if not entered. Immutable. |
| Parent Work Order | `PARENT_ID` | Parent work order `ORDER_NO` (self FK). Semi-product children have the finished good's ORDER_NO. Root is NULL. |
| Co-generation Root | `ROOT_ORDER_NO` | Top-level ORDER_NO of BOM auto-expansion group. Root itself is NULL. |
| Production Plan No. | `PLAN_NO` | Linked `PROD_PLANS.PLAN_NO` (optional). On cancel, decrements the plan's `ORDER_QTY`. |
| Item Code | `ITEM_CODE` | References `PART_MASTERS.ITEM_CODE` (required). Validates item existence and tenant match on creation. |
| Line | `LINE_CODE` | Production line (optional). |
| Routing Code | `ROUTING_CODE` | Auto-resolved from item (`ROUTING_GROUPS`, useYn='Y'). No direct input. |
| Process | `PROCESS_CODE` | Representative process. Auto-inherited from routing's first SEQ if not set. Equipment reset on process change. |
| Equipment | `EQUIP_CODE` | Work equipment (optional). Usually NULL at creation, assigned later. |
| Plan Qty | `PLAN_QTY` | int. Required, ≥ 1. Child (semi-product) quantity = `ceil(parent qty × BOM qty per)`. |
| Good Qty | `GOOD_QTY` | int (default 0). Result aggregation (sum of PROD_RESULTS). Confirmed on completion. |
| Defect Qty | `DEFECT_QTY` | int (default 0). Result aggregation. |
| Plan Date | `PLAN_DATE` | date (nullable). Default filter for list query. NULL entries always shown regardless of range filter. |
| Start Time | `START_TIME` | timestamp. Recorded once on first start. |
| End Time | `END_TIME` | timestamp. Recorded on complete/cancel. |
| Priority | `PRIORITY` | int (default 5). 1(highest)~10(lowest). List sorted by PRIORITY ASC, PLAN_DATE ASC, CREATED_AT DESC. |
| Status | `STATUS` | varchar2 20 (default WAITING). Common code `JOB_ORDER_STATUS`. Values: WAITING/RUNNING/HOLD/DONE/CANCELED. |
| Customer PO No. | `CUST_PO_NO` | Corresponding customer PO (optional). |
| Remark | `REMARK` | varchar2 500. On HOLD, injects `[HOLD] previous state:{state}` prefix to preserve return status. |
| ERP Sync | `ERP_SYNC_YN` | 'Y'/'N' (default N). Among DONE entries, those with 'N' are shown as unsynced list for ERP. |
| Multi-tenancy | `COMPANY`, `PLANT_CD` | Company/plant scope. Standard values `COMPANY='40'`, `PLANT_CD='1000'`. Validates tenant match with item and routing. |
| Creator/Modifier | `CREATED_BY`, `UPDATED_BY` | Audit columns. |
| Created/Updated At | `CREATED_AT`, `UPDATED_AT` | timestamp (DEFAULT SYSTIMESTAMP). |

> Child work orders (auto-generated semi-products) have `REMARK` set to `[Auto-generated] Semi-product of {parent ORDER_NO}`.

## Generation Rules
- Generation: `nextJobOrderNo()` → channel `JOB_ORDER` (Oracle SEQUENCE-based, shares `QueryRunner` within transaction).
- Format: `W + YYMMDD + - + 3-digit daily sequence`. Example: `W260519-001`. Daily reset at 00:00 (DBMS_SCHEDULER).
- Detail (part-level) code convention: `FullCode-PartNumber` (e.g., `W260519-001-HNS-A-R1`). Serial replaces leading `W` with `S`.
- Single source: `docs/standards/numbering-rules.md`.

## State Transition Logic
Transitions only via dedicated APIs. `PUT /:id/status` (direct change) is always rejected (prevents state jump bypass).

| Transition | API | Allowed Pre-state | Processing |
|------|------|------|------|
| Start | `POST /:id/start` | WAITING → RUNNING | Records START_TIME first time. If `FG_BARCODE_ISSUE_TIMING=PRE_ISSUE`, issues FG_LABELS for plan qty. |
| Hold | `POST /:id/hold` | WAITING/RUNNING → HOLD | Injects `[HOLD] previous state:{state}` into REMARK (for restore). Blocks result registration and shipping. |
| Hold Release | `POST /:id/hold-release` | HOLD → previous state | Parses previous state from REMARK and restores, removes prefix. |
| Complete | `POST /:id/complete` | RUNNING → DONE | Confirms GOOD_QTY/DEFECT_QTY from PROD_RESULTS sum, records END_TIME (transaction). Ends regardless of remaining qty. |
| Cancel | `POST /:id/cancel` | WAITING/HOLD → CANCELED | Blocked if any result exists. If linked to PLAN_NO, decrements plan's ORDER_QTY, records END_TIME. |

- **Edit (PUT)**: DONE/CANCELED cannot be edited. Direct `status` field edit also rejected. Re-queries ROUTING_CODE on itemCode change.
- **Delete (DELETE)**: RUNNING cannot be deleted (other statuses allow hard delete).

## BOM Auto-Expansion Logic (on Creation)
1. If `autoCreateChildren !== false` (default ON), starts recursive expansion after parent save.
2. Query `BOM_MASTERS`(parentItemCode=current item, useYn='Y') for child items.
3. Only children with `PART_MASTERS.ITEM_TYPE='SEMI_PRODUCT'` are created as work orders.
4. Child PLAN_QTY = `ceil(parent PLAN_QTY × BOM.QTY_PER)`. PARENT_ID=parent, ROOT_ORDER_NO=root.
5. Repeat steps 2-4 recursively for children. Circular reference (ancestor path tracking)·depth 50 backstop prevents infinite loops.

## Prerequisites (Master·Common Code)
- Common codes: `JOB_ORDER_STATUS`(status badge/filter), `JOB_ORDER_TYPE`(type: NORMAL/REWORK/SAMPLE/TRIAL).
- Prerequisite masters: Item Master (`PART_MASTERS`), Routing (`ROUTING_GROUPS`/`ROUTING_PROCESS`), BOM (`BOM_MASTERS`), Equipment·Line.
- System setting: `FG_BARCODE_ISSUE_TIMING`(`ON_INSPECT` default / `PRE_ISSUE` batch-issues barcodes on start·pre-issuance).

## Operating Procedure
1. Register item, routing, and BOM first (without routing, process/equipment cannot be auto-inherited).
2. Create work order (item·plan qty·plan date required). Auto-expansion creates semi-products simultaneously.
3. Proceed in order: Start → (register production results) → Complete. Use hold/release as needed.
4. Check completed entries in ERP unsynced list (`GET /erp/unsynced`) and process sync (`POST /erp/mark-synced`).

## Permissions
Production manager (create/edit/delete·state transitions). General users: view. (View/manage only within multi-tenancy scope.)

## Troubleshooting
| Symptom | Cause | Action |
|------|------|------|
| Start button disabled | Status not WAITING | Only WAITING can start. Hold entries need hold-release first. |
| Cannot complete (400) | Status not RUNNING | Start (RUNNING) first, then complete. |
| Cancel rejected (400) | Results exist | Delete results first, or complete instead. |
| Edit rejected (400) | DONE/CANCELED status or direct status change attempted | Completed/cancelled cannot be edited. Use dedicated APIs for status. |
| Delete rejected (400) | RUNNING status | Cannot delete in progress (complete/cancel first). |
| Routing/process not auto-filled | Item routing not registered (no useYn='Y') | Register routing group or set process manually. |
| Equipment list empty | No equipment mapped to selected process | Register equipment for that process in Equipment Master. |
| Semi-product work orders not created | BOM child not SEMI_PRODUCT/BOM not registered/checkbox off | Check BOM·item type, ensure auto-create checkbox ON. |
| Duplicate number (409) | Same ORDER_NO exists | Use auto-generation (avoid manual input). |
| Not showing in list | Outside plan date filter range | Expand filter (NULL plan date always shown). |

## Data & Integration
- Table: `JOB_ORDERS`(+ self-reference PARENT_ID/ROOT_ORDER_NO)
- References: `PART_MASTERS`(ITEM_CODE), `ROUTING_GROUPS`/`ROUTING_PROCESS`(ROUTING_CODE), `BOM_MASTERS`(expansion), `PROD_PLANS`(PLAN_NO)
